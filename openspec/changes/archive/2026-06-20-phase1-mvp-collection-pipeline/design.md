## Context

千机笔系统已有 FastAPI 主服务（ArticleGeneratorService）、LLM Gateway、HotspotCrawler、Vue 3 前端。现有技术栈包括 SQLAlchemy ORM、Celery 任务队列、JWT 认证。Phase 1 需要在现有系统基础上新增数据采集管道，所有新增组件与现有系统集成但保持独立（不修改现有表结构和 API）。

完整技术设计文档见 `docs/superpowers/specs/2026-06-20-phase1-mvp-design.md`。本文档聚焦关键架构决策及其理由。

## Goals / Non-Goals

**Goals:**
- 建立赛道分类体系（一级+二级赛道）
- 实现公众号注册和自动信息获取（searchbiz + 链接解析）
- 实现公众号文章采集执行引擎（凭证管理 → 任务调度 → 文章拉取 → 去重入库）
- 提供素材文章浏览功能（列表筛选 + 原文/Markdown 切换）
- 采集引擎具备防风控能力（低频间隔 + 抖动 + 串行）

**Non-Goals:**
- 不做图片下载和转存（三方文章图片参考价值低）
- 不做实时采集（低频场景，每天一次即可）
- 不做知识库提取（Phase 2）
- 不做论点风暴集成（Phase 3）
- 不修改现有 articles 表结构

## Decisions

### 1. 采集引擎：复用 Celery 而非新建独立服务

**选择:** 在现有 Celery app 中新增 collector tasks，使用 Celery Beat 做调度

**替代方案:**
- APScheduler in-process: 简单但缺乏任务持久化和重试能力
- 独立 Python 脚本 + cron: 运维复杂，状态追踪困难
- 新建微服务: 过度设计，Phase 1 不需要

**理由:** 现有系统已部署 Celery（生成任务、蒸馏任务），复用基础设施减少运维成本。Celery Beat 原生支持 cron 调度，Worker 提供任务隔离和重试。

### 2. 解析策略：延迟解析 (Lazy Parse)

**选择:** 采集时只存 raw_html，Markdown 转换推迟到用户查看或 AI 调用时触发

**理由:**
- 采集是低频操作（每天一次），但解析是高频操作（每次查看）
- 99% 的采集文章可能永远不会被打开，提前解析浪费 CPU
- 用户反馈：图片处理和完整解析不是采集阶段的刚需
- 解析结果缓存到 content_markdown 字段，后续访问零成本

### 3. 频次控制：防风控优先于采集速度

**选择:** 全局请求间隔 ≥15s + 随机抖动，同域 ≥20-30s，单号 ≥24h，并发 ≤2

**理由:**
- 采集目标是微信公众号后台接口，非官方公开 API，需模拟人类行为
- 每天采集一次，对速度无要求，风控是首要考量
- 过于激进的频率会导致 IP/账号被限制，得不偿失
- 前期手动填入凭证，凭证宝贵不可再生

### 4. 凭证管理：手动填入 + 自动检测

**选择:** 前期由用户手动填入 token+cookie，系统提供定时健康检测和任务前预检

**理由:**
- 微信公众号后台无 OAuth/API Key 机制，只能使用登录态凭证
- 凭证有有效期，需要持续监控状态
- 自动获取凭证（模拟登录）风险极高且不稳定，不做

### 5. mp_materials 与现有 articles 表分离

**选择:** 新建 mp_materials 表存储采集素材，不复用现有 articles 表

**理由:**
- articles 表存储 AI 生成的文章（需要 account_id/hotspot_id FK），语义不同
- mp_materials 存储外部采集的原始文章（需要 original_url/content_hash 去重字段）
- 分离避免字段冗余和语义混淆
- 未来素材中心、知识库都基于 mp_materials 构建

### 6. 去重策略：URL 唯一索引 + 内容哈希

**选择:** original_url 设唯一索引（第一道防线），SHA256(raw_html) 做内容去重（第二道防线）

**理由:**
- URL 去重最快（数据库级别，INSERT 时自动检测）
- 内容哈希防止同一篇文章通过不同 URL 重复入库（如带参数 URL、短链展开）
- 标题相似度为第三道防线，Phase 1 暂不实现（成本高、收益低）

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 微信公众平台接口变更（appmsg 参数或返回格式变化） | MpClient 封装所有 HTTP 调用，变更仅影响单个模块 |
| 凭证过期导致采集中断 | 每 6 小时自动检测 + 任务前预检 + 前端状态可视化 |
| raw_html 存储占用大 | 单篇文章 HTML 通常 50-200KB，1000 篇约 50-200MB，可接受。后续可考虑压缩 |
| 采集速度慢（15s 间隔） | 既定设计目标：低频场景不追求速度 |
| 并发限制导致任务排队 | 每天一次，排队数小时可接受 |
